What is SDLC?

Software Development Life Cycle (SDLC) is a structured process used to design, develop, test, and deploy high-quality software efficiently and systematically.

Purpose of SDLC:

Ensure high-quality software delivery
Reduce development risks and cost
Improve project planning and control
Ensure clear communication among teams

**QA Perspective:

QA ensures that each SDLC phase is testable and defects are identified early.

Phases of SDLC:

1. Requirement Gathering & Analysis
Collect business needs from stakeholders
Document requirements
2. System Design
Create architecture and system design
Define database and UI design
3. Implementation (Coding)
Developers write code based on design documents
4. Testing
QA team performs testing to find defects
Includes functional and non-functional testing
5. Deployment
Software is released to production environment
6. Maintenance
Bug fixing and system updates after release

QA Role:

QA is involved from requirement phase to maintenance support testing.

SDLC Models:

🔷 Waterfall Model

A linear model where each phase is completed before moving to next.

Advantages:

Simple and easy to understand
Well-structured phases

Disadvantages:

Not flexible for changes
Late testing phase

🔷 Agile Model

Iterative model where development happens in small cycles (sprints).

Advantages:

Flexible to changes
Early delivery of working software

Disadvantages:

Less documentation
Requires high customer involvement

🔷 V-Model

Testing phase runs parallel with development phase.

Advantages:

Early defect detection
Strong testing focus

Disadvantages:

Rigid structure
Not suitable for complex changes

**QA Perspective:

Agile = continuous testing
Waterfall = testing at final stage
V-Model = early QA involvement

Requirement Gathering:

Requirement gathering is the process of collecting and analyzing user and business needs before development starts.

📌 Sources:
Stakeholders
Business analysts
End users

📌 Activities:
Interviews
Surveys
Document analysis
Meetings

**QA Role:

QA reviews requirements to ensure they are-

Clear
Testable
Complete

Types of Requirements:

🔷 Functional Requirements

Describe what the system should do.

Example:

User login system
Payment processing

🔷 Non-Functional Requirements

Describe how the system performs.

Example:

Performance (load time < 2 seconds)
Security (data encryption)
Usability

🔷 Business Requirements

High-level business goals.

Example:

Increase customer engagement
Improve sales conversion

**QA Perspective:

QA writes test cases based on both functional and non-functional requirements.
